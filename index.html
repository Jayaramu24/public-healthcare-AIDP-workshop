Business Context

MPHA turns healthcare operating pressure into trusted insights and guided action

Metro Public Health Authority needs to improve claims transparency, reduce denial leakage, monitor disbursements, track membership and provider accreditation, and turn policy guidance into faster operational decisions. This execution flow shows how Oracle AI Data Platform, Autonomous AI Lakehouse, Oracle Analytics Cloud, machine learning, and AI agents work together to deliver that operating model.

MPHA business problems

  • Claims transparency and denial leakage across districts, programs, and claim types.
  • Disbursement visibility where submitted amounts do not convert into paid amounts.
  • Membership, provider accreditation, service-area, and facility-pressure accountability.
  • Policy guidance that is difficult to operationalize from long playbook documents.

Workshop capabilities

  • OAC Claims Executive Overview with KPIs, trends, hotspots, maps, and Assistant responses.
  • Claims star schema in Autonomous AI Lakehouse as the trusted Gold serving layer.
  • Round 2 JSON and spatial context extension for richer district-level Claims insights.
  • Denial risk scoring to prioritize claims that need review or intervention.
  • Claims and Policy Copilot that combines SQL analytics with RAG over the MPHA playbook.

Lab 1

Set up the shared OCI, AIDP, AI Lakehouse, and OAC environment

Admin-led · 45-60 minutes
Admin step

The facilitator prepares the environment once so first-time participants are not distracted by tenancy, policy, user, bucket, catalog, and service provisioning details. Do not move to Lab 2 until these setup steps are complete and visible to the participants.

Admin ownership Compartment, Object Storage, AIDP, AI Lakehouse, OAC, catalog, volume, and policy setup are admin tasks. Participants should receive a ready workspace and start active work from notebook upload and execution.

Admin setup sequence

  1. Create or select the OCI compartment. In the OCI Console, open Identity & Security - Compartments. Create a compartment such as MPHA_AIDP_WORKSHOP or select the compartment assigned by the tenancy admin. Use this same compartment for Object Storage, AIDP, Autonomous AI Lakehouse, OAC, Generative AI Agents, and policies.
  2. Add the required standard policies. Apply the AIDP standard policies generated during AIDP setup. Confirm the facilitator group can manage AIDP workspaces, clusters, catalogs, volumes, Object Storage buckets, AI Lakehouse resources, OAC connections, and Generative AI Agent resources where optional labs are included.
  3. Create the Object Storage bucket. Open Storage - Buckets, select the workshop compartment, and create a bucket such as mpha-workshop-bucket. Keep the default storage tier unless your tenancy standard says otherwise.
  4. Create the Object Storage folder structure. Open the bucket and create the prefixes listed in the folder contract below. If the console does not show a create-folder action, upload a small placeholder file into each prefix.
  5. Upload the raw workshop assets. Upload the five CSV files under mpha/raw/, the JSONL file under mpha/raw_json/, the GeoJSON file under mpha/raw_spatial/, and the MPHA playbook document under mpha/documents/.
  6. Create or confirm the AIDP instance. Open Analytics & AI - AI Data Platform, create or reuse the workshop AIDP instance, and confirm the standard policies are applied successfully.
  7. Create the AIDP workspace and Spark cluster. Create a workspace such as mpha-core-workshop, create a Spark cluster such as mpha-spark-cluster, and confirm participants can attach that cluster to notebooks.
  8. Create the standard catalog, schema, and external volume. In AIDP Master Catalog, create or confirm the standard catalog and schema used for the workshop landing area. Create external volumes that point to the Object Storage bucket and folder prefixes.
  9. Create the external AI Lakehouse catalog. Create the AIDP external catalog connection to Autonomous AI Lakehouse, test connectivity, and confirm the participant schema is visible.
  10. Create the AI Lakehouse target schema and user. In Autonomous AI Lakehouse Database Actions, create the workshop user or schema, set quota high enough for loading, REST-enable the user if needed, and run the Claims star schema table creation SQL.
  11. Create the workspace notebook folders. In AIDP, create folders such as O1_Bronze, O2_Silver, 03_Gold, 03a_GoldAILHLoad, 04_ML, 05_Agent, and Shared. Participants will upload the downloaded notebooks into these folders during the labs.
  12. Create or confirm OAC. Provision Oracle Analytics Cloud or confirm the shared OAC instance is ready. Do not build the OAC dataset until after the Claims star schema has been loaded and validated in Lab 3.

Operational checkpoint - create the AIDP Workflow for incremental medallion runs

After the notebook sequence is clear, review the validated AIDP Workflow version of the same flow. This is the operational view of the workshop pipeline: every task has a file, compute target, timeout, dependency, and a successful run trail.

Workflow step 1

Open Workflow and create the job

In AIDP Workbench, open the shared workshop workspace, click Workflow, and create a workflow named MPHA_INCREMENTAL_MEDALLION_FLOW.

Workflow step 2

Configure the Bronze ingest task

Rename the first task to bronzeingest. Select the task type that matches the uploaded asset. In the validated run, Bronze used a Python task pointing to /Workspace/O1_Bronze/aidp_bronze_pyspark.py.

  • Select the Bronze file from the O1_Bronze folder.
  • Select compute E2EAIDPIndustrydemos.
  • Set an execution timeout before adding the next task. The validated workflow uses 30 minutes.
Workflow step 3

Add Silver after Bronze

Add silverrefine after the Bronze task has a timeout value. Set the dependency to bronzeingest, choose the Silver notebook or Python file, select the same compute, and set a timeout.

Workflow step 4

Add Gold staging and AI Lakehouse load tasks

Add goldstage after silverrefine. Then add lakehouseload after goldstage. For both tasks, choose the file from the correct workspace folder, select compute, and set a timeout.

Workflow step 5

Review and run the workflow

Before running, confirm the chain is bronzeingest to silverrefine to goldstage to lakehouseload. Run the workflow and watch the graph view.

Workflow step 6

Confirm task progress and final success

Open task details as the workflow runs. The Bronze output proves the raw files were written to the shared volume. The final lakehouseload task confirms the Claims star schema load.

Optional Lab 5

Track, train, register, and score a Claims denial risk model

Optional extension · 60 minutes
Optional extension

This optional lab turns the Claims star schema into a predictive review-prioritization use case. Participants upload and run an AIDP ML notebook, review the tracked run in AIDP Experiments, register the selected model in the catalog, and publish batch scores that can be added to AI Lakehouse or OAC as a follow-on analytics extension.

Preview feature note AIDP Experiments and Models are preview capabilities in the July 2026 Oracle documentation. Use this lab as an optional extension and confirm compute compatibility before delivery. The flow below was validated in the workshop AIDP instance on July 7, 2026.

Business use case

MPHA wants to predict which district, coverage program, and claim-type combinations are most likely to show high denial risk in the next review cycle. Claims operations can then prioritize documentation checks, provider follow-up, policy remediation, and payment leakage review before the backlog grows.

Official AIDP references used

  • AIDP Experiments: experiment tracking, MLflow autologging, run comparison, reproducibility, retraining, and registering a model from a run.
  • AIDP Models: model registry, versions, metrics, artifacts, lineage, and batch inference usage.
  • Customer-managed MLflow servers: optional external tracking server pattern and its tradeoff with built-in AIDP Experiments and Models integration.

Optional extension

This lab builds the MPHA Claims and Policy Copilot in Oracle AI Data Platform. The copilot uses one supervisor agent, one SQL executor for the Claims star schema, and one RAG executor that consumes the mphapolicy knowledge base prepared in Lab 6A.

Admin and participant split Admins prepare the AI compute, the AI Lakehouse external catalog, and the required access policies. Participants use the mphapolicy knowledge base from Lab 6A and then build and test the agent flow.

Business scenario

Claims leaders want a single copilot that can answer what is happening in claims operations and what the operating playbook recommends next. SQL answers come from the governed Claims star schema. Policy guidance comes from the MPHA Winter Respiratory Response Playbook.

Question typeExecutor pathExample
Claims metricSUPERVISOR_AGENT_1 -> AGENT_1 -> SQL_1Which districts have the highest denial rate?
Policy or playbookSUPERVISOR_AGENT_1 -> AGENT_2 -> RAG_1What does the playbook recommend for high occupancy?
Combined actionSUPERVISOR_AGENT_1 -> AGENT_1 -> AGENT_2 -> synthesisWhere are the denial hotspots and what playbook action should operations take?

Prerequisites

  1. Admin step: Validate the Claims star schema. Confirm mpha_fact_claims_monthly, mpha_dim_date, mpha_dim_district, mpha_dim_coverage_program, and mpha_dim_claim_type are populated in AI Lakehouse.
  2. Admin step: Confirm the external catalog. Confirm AIDP can read the AI Lakehouse catalog and schema, for example goldailh.e2eaidpuser.
  3. Participant step: Confirm the playbook knowledge base from Lab 6A. The MPHA playbook must be indexed as e2eindustrydemos.default.mphapolicy or the equivalent workshop knowledge base.
  4. Admin step: Confirm AI compute. Create or reuse an AI compute such as AIComputeForAgents and make sure it is active before testing.
  5. Participant step: Open AIDP Workbench. Use the shared workspace E2EAIDPIndustryDemos and open Agent flows.

Task 1 - Create or open the copilot flow

  1. Open Agent flows from the AIDP workspace navigation.
  2. Click Create to create a new flow, or open the prepared MPHA_Claims_Policy_Copilot flow.
  3. Name the flow MPHA_Claims_Policy_Copilot.
  4. Use visual authoring mode so participants can see the supervisor, executor agents, and tools on one canvas.

Task 2 - Configure the supervisor agent

  1. Select the supervisor node and name it SUPERVISOR_AGENT_1.
  2. Select the approved generative model. In the validated workshop run, the model is google.gemini-2.5-flash.
  3. Connect the chat trigger to the supervisor.
  4. Connect the supervisor to both executor agents: AGENT_1 for SQL and AGENT_2 for RAG.
  5. Add routing instructions that force the supervisor to call executor agents instead of answering directly.
Available executor agents:
- AGENT_1: structured MPHA claims analytics from the AI Lakehouse Claims star schema.
- AGENT_2: document-grounded MPHA policy, playbook, accreditation, prior authorization, denial remediation, disbursement, membership, and facility-pressure guidance.
- For structured claims questions, call AGENT_1.
- For document or policy questions, call AGENT_2.
- For combined data-plus-policy questions, call AGENT_1 first, then AGENT_2, then synthesize both results.
- Do not call SQL_1 or RAG_1 directly from the supervisor.

Task 3 - Configure the Claims SQL executor

  1. Add an Agent node and name it AGENT_1.
  2. Describe it as the Claims SQL executor for denial-rate, payment, district, program, claim type, and processing-day metrics.
  3. Use the same approved model as the supervisor.
  4. Connect AGENT_1 to SQL_1.
  5. Add mandatory behavior that forces AGENT_1 to call SQL_1 before answering any structured claims question.

Task 4 - Configure the Claims star schema SQL tool

  1. Add a SQL tool and name it SQL_1.
  2. Use table or SQL-query mode depending on the AIDP environment available to the workshop team.
  3. Point the tool to the Claims star schema in AI Lakehouse.
  4. Expose only the fields required for the workshop: service month, district, coverage program, claim type, claims submitted, denied claims, denial rate, submitted amount, paid amount, and average processing days.
  5. Create an input such as TOP_N so the executor can return the requested number of hotspots.
SELECT
  d.full_date AS service_month,
  di.district_name,
  p.coverage_program,
  c.claim_type,
  SUM(f.claims_submitted) AS claims_submitted,
  SUM(f.denied_claims) AS denied_claims,
  ROUND((SUM(f.denied_claims) * 100.0) /
        CASE WHEN SUM(f.claims_submitted) = 0 THEN NULL ELSE SUM(f.claims_submitted) END, 2) AS denial_rate_pct,
  ROUND(SUM(f.total_submitted_amount), 2) AS total_submitted_amount,
  ROUND(SUM(f.total_paid_amount), 2) AS total_paid_amount,
  ROUND(AVG(f.avg_processing_days), 1) AS avg_processing_days
FROM mpha_fact_claims_monthly f
JOIN mpha_dim_date d ON f.service_month_date_key = d.date_key
JOIN mpha_dim_district di ON f.district_key = di.district_key
JOIN mpha_dim_coverage_program p ON f.program_key = p.program_key
JOIN mpha_dim_claim_type c ON f.claim_type_key = c.claim_type_key
GROUP BY d.full_date, di.district_name, p.coverage_program, c.claim_type
ORDER BY denial_rate_pct DESC, denied_claims DESC
FETCH FIRST :TOP_N ROWS ONLY

Task 5 - Configure the Policy RAG executor

  1. Add a second Agent node and name it AGENT_2.
  2. Describe it as the Policy RAG executor for playbook, accreditation, prior authorization, denial remediation, disbursement, membership, and facility-pressure guidance.
  3. Use the same approved model as the supervisor.
  4. Connect AGENT_2 to RAG_1.
  5. Add instructions that force AGENT_2 to call RAG_1 and cite retrieved evidence.

Task 6 - Configure the MPHA playbook RAG tool

  1. Add a RAG tool and name it RAG_1.
  2. Select the knowledge base e2eindustrydemos.default.mphapolicy, or the equivalent knowledge base created for this workshop.
  3. Set the retrieval limit to return enough chunks for policy synthesis, while keeping responses concise.
  4. Use a natural-language input query so AGENT_2 can pass policy, playbook, and operational-pressure questions to the tool.
  5. Instruct downstream answers to include source paths, chunk identifiers, or section references when available.

Task 7 - Deploy the flow to AI compute

  1. Confirm the canvas contains the chat trigger, supervisor, SQL executor, RAG executor, SQL tool, and RAG tool.
  2. Open the compute selector and choose AIComputeForAgents or the AI compute prepared by the facilitator.
  3. Deploy the flow.
  4. Wait until the banner shows the AI compute as active and the page shows Undeploy, which confirms the flow is deployed.

Task 8 - Open Playground and start a clean session

  1. Switch from Development to Playground.
  2. Select SUPERVISOR_AGENT_1 as the test target.
  3. Create a new session for the participant run so earlier tests do not affect the conversation.
  4. Keep the test prompt focused on the workshop scope.
